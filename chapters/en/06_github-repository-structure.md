## Infrastructure Repositories

### `Wamocon/github_workflow`

The central CI/CD repository of the organisation. Contains reusable GitHub Actions workflows
that all projects reference:

```
github_workflow/
├── .github/workflows/
│   ├── pr-checks.yml           # TypeScript + ESLint validation (reusable)
│   ├── pr-autofix.yml          # Auto-fix lint & formatting errors (reusable)
│   ├── deploy-preview.yml      # Vercel preview deployment (reusable)
│   └── deploy-production.yml   # Vercel production deployment (reusable)
├── examples/
│   ├── caller-pr-pipeline.yml  # Template for PR workflow in own repo
│   └── caller-production-deploy.yml  # Template for deploy workflow
└── docs/
    ├── workflow-guide.md
    ├── supabase-vercel-linking.md
    ├── database-schema-migrations.md
    ├── mcp-setup.md
    ├── github-copilot-guide.md
    └── tips-and-best-practices.md
```

### `Wamocon/localSupabaseDB`

Template repository for local Supabase development environments.
Each Next.js app gets its own copy if needed, to allow development locally without Supabase Cloud.

### `Wamocon/template_repo`

The base template for all new WAMOCON apps. Contains full configuration for:

- Next.js 16 App Router
- Supabase integration
- GitHub Actions CI/CD
- Copilot agents and instructions

## Visibility and Access

- All app repositories are **internal** by default (visible to org members only)
- Landing pages may be public depending on requirements
- Repos are temporarily made **public** for the initial Vercel deployment, then set back to internal

## Branch Strategy

### Default: main / dev

The standard approach for all WAMOCON apps:

```
main   ← Production branch, always deployable, no direct commits
dev    ← Development branch, active work
```

- Development happens on `dev`
- PRs run from `dev` to `main` after review and tests
- Direct commits to `main` are not allowed

### Topic-Specific Branching (occasional)

For larger features or parallel development, additional branches can be used:

```
feature/auth       ← New feature
fix/bugname        ← Bug fix
hotfix/critical    ← Urgent production fix
```

These branches are created from `dev` and merged back to `dev`.

**Rule:** Never commit directly to `main`, always go through `dev` and then a PR.

## Naming Conventions

WAMOCON repositories use two writing styles:

- **`snake_case`** (lowercase, words separated by underscore): e.g. `backup_planner`, `carman_lp`, `daily_echo`
- **`camelCase`** (capitalised first letter): e.g. `localSupabaseDB`, `AppMonitor`, `TeamRadar`

Special cases and exceptions are documented in the App Catalogue (Chapter 9).

## Repository Structure (Standard App)

All app repos are based on the `template_repo` template and have this structure:

```
<app-name>/
├── .github/
│   ├── agents/                 # AI personas (@planner, @developer, @reviewer)
│   ├── instructions/           # File-specific Copilot rules
│   ├── workflows/
│   │   ├── pr-pipeline.yml     # Calls central workflow
│   │   └── deploy.yml          # Calls central deploy workflow
│   └── copilot-instructions.md # Global Copilot rules
├── src/
│   └── app/                    # Next.js App Router
├── supabase/
│   └── migrations/             # SQL migration files
├── tests/                      # Playwright E2E tests
├── legal-docs/                 # Imprint, privacy policy, T&Cs
├── .env.example                # Template for environment variables
├── HOWTO.md                    # Setup & deployment guide (DE/EN)
├── AGENTS.md                   # Copilot agents documentation
├── package.json
├── tsconfig.json
├── next.config.ts
└── eslint.config.mjs
```
